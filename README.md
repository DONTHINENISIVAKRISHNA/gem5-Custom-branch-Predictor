# gem5-Custom-branch-Predictor
In my Computer Architecture course , I implemented a custom GShare-style branch predictor from scratch in C++ inside the gem5 ARM ISA simulator running an O3 out-of-order CPU pipeline. The predictor uses a NAND operation,instead of the standard XOR used in classic GShare, to combine the lower bits of the branch program counter with the global history register to index a pattern history table. I implemented five core functions: lookup(), which computes the NAND index and reads two counters  takenCounters and notTakenCounters,predicting taken only when the taken counter is strong and the not-taken counter is weak simultaneously; update(), which strengthens the correct counter on a correct prediction and updates the opposite counter on a misprediction, skipping counter updates entirely when the pipeline is squashed to prevent wrong-path data from corrupting the predictor state; squash(), which restores the global history register to its pre-prediction state when a misprediction is detected — critical because a corrupted history would contaminate all future predictions; uncondBranch(), which updates history with taken for unconditional branches since no prediction is needed; and btbUpdate(), which clears the LSB of history on a BTB miss to force a not-taken prediction when the branch target address is unknown. The global history register shifts left on every branch outcome ,inserting 1 for taken and 0 for not taken,masked to 10 bits tracking the last 10 branch outcomes. Running dijkstra_small and qsort_small benchmarks I achieved a 91% BTB hit ratio and IPC of 0.825, with 85% conditional branch prediction accuracy across 19.5 million branch lookups. The NAND operation produces a biased index distribution, mapping more branch contexts to higher PHT indices compared to XOR's uniform distribution,increasing aliasing between unrelated branches sharing the same counter, which explains why my accuracy was slightly lower than standard GShare's typical 87-91%. This project gave me deep understanding of how branch mispredictions cause pipeline flushes that directly reduce IPC, exactly the kind of HW/SW interaction that matters in high-performance processor design."
